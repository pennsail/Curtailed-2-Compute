# Data Center Financial Model
This model's primary goal is to perform a "reverse-engineering" analysis: instead of forecasting a rent and seeing the return, it finds the **minimum viable rent** a project must charge to achieve a **target Levered IRR (e.g., 15%)**.

Crucially, it only finds solutions that are "bankable," meaning they also satisfy a lender's constraints for Debt Service Coverage (DSCR) and an equity investor's minimum yield.

## How to Use This Model

This notebook is run as a workflow:

1.  **Configure Scenarios (`Cell 3`):** This is the main input cell. You define the global assumptions and the specific parameters for each investment case.
2.  **Run the Solver (`Cell 5`):** This cell runs the main analysis, finding the minimum rent for each scenario and printing a summary table.
3.  **Run Sensitivity Analysis (`Cell 6`):** This cell takes the results from `Cell 5` and runs sensitivity checks on key variables, plotting the impact on profitability.

## 1. Key Configuration Parameters (`Cell 3`)

These dictionaries are the primary inputs for the entire model.

### `BASE_PARAMS`
Global assumptions that apply to all scenarios over the 25-year life of the model.

* `scale_mw`: The IT load capacity of the data center (e.g., `20.0` MW).
* `years`: The total analysis period (e.g., `25` years).
* `debt_term_years`: The full amortization period of the loan.
* `discount_rate`: The "hurdle rate" used to calculate Net Present Value (NPV).
* `rent_escalator`: The annual percentage increase in rent (e.g., `0.025` for 2.5%).
* `opex_escalator`: The annual percentage increase in operating costs (inflation) (e.g., `0.03` for 3.0%).

### `SOLVER_TARGETS`
The "pass/fail" criteria that define a "bankable" project.

* `TARGET_IRR`: The minimum Levered IRR the equity investor must achieve (e.g., `0.15` for 15%).
* `DSCR_RAMP`: A tuple defining the minimum Debt Service Coverage Ratio (DSCR) for `(Year 1, Year 2, Year 3+)`. This models a lender's covenants.
* `MIN_YIELD`: The minimum required `steady_state_yield` (Final Year NOI / Total CapEx).

### `SCENARIO_CONFIGS`
A dictionary containing the specific, unique parameters for each investment case (e.g., "A\_Metro", "B\_Rural\_Flex").

* `total_capex`: The all-in cost to build the facility.
* `opex_y1`: The non-power operating cost for the *first year* (includes staffing, maintenance, etc.).
* `initial_utilization`: The datacenter's occupancy/utilization in Year 1 (e.g., `0.25` for 25%).
* `target_utilization`: The stabilized, long-term occupancy (e.g., `0.85` for 85%).
* `lease_up_years`: The number of years it takes to ramp from `initial_utilization` to `target_utilization`.
* `service_revenue_pct`: Additional revenue from managed services, as a percent of base rent.
* `capex_incentive_pct`: Any upfront capital incentive or tax credit, as a percent of `total_capex`.
* `terminal_multiple`: The "exit multiple" used to calculate the sale price at the end of 25 years (Sale Price = Final Year NOI * `terminal_multiple`).
* `debt_ratio`: The loan-to-cost ratio (e.g., `0.55` for 55% debt).
* `debt_rate`: The interest rate on the loan (e.g., `0.075` for 7.5%).
* `io_years`: The number of "interest-only" payment years at the start of the loan, set to match the `lease_up_years`.
* `bess_revenue_y1`: The specific Year 1 revenue generated by the Battery Energy Storage System (BESS) from arbitrage or grid services.

## 2. Core Model Engine (`Cell 2`)

### `run_model(cfg)`
This is the central engine of the entire notebook. It takes a single configuration dictionary (`cfg`) and builds a 25-year financial pro-forma. It calculates the Net Operating Income (NOI), Terminal Value (TV), unlevered (all-cash) cash flows, and levered (with-debt) cash flows. It returns a dictionary containing all key metrics (IRR, NPV, DSCR) and the underlying data series.

### Model Helper Functions

`run_model` uses these functions to build its pro-forma:

* **`lease_up(initial_u, target_u, years_to_target, T)`**
    * **Logic:** Models the S-curve of a new facility's occupancy. It builds an array representing utilization, starting at `initial_u`, ramping linearly to `target_u` over `years_to_target`, and then holding flat at `target_u` for the remaining life of the project.

* **`build_revenue(...)`**
    * **Logic:** Creates the "revenue stack" for each year. It calculates the `base_rent` (based on `util_curve` and `rent_escalator`), then adds `service_revenue` and `bess_revenue` on top.

* **`build_opex(opex_y1, opex_escalator, T)`**
    * **Logic:** Creates the operating cost series. It starts with `opex_y1` and grows it by the `opex_escalator` rate each year.

* **`debt_service_series(...)`**
    * **Logic:** Calculates the project's loan payments. It returns the total `debt`, the "interest-only" payment (`io_pay`) for the first `io_years`, and the fully amortizing (principal + interest) payment (`amort_pay`) for the rest of the loan.

## 3. Solver & Bankability Functions (`Cell 4`)

### `find_min_rent_for_target_irr(...)`
This is the core analysis function. It uses a bisection (binary search) algorithm to find the *lowest possible rent* that allows the project to hit the `TARGET_IRR` *and* pass all `meets_bankability` checks.

### Bankability Helper Functions

* **`meets_bankability(res)`**
    * **Logic:** A master "pass/fail" check. It returns `True` only if a project result passes all three tests:
        1.  The IRR is a valid number.
        2.  The `steady_state_yield` is $\ge$ `MIN_YIELD`.
        3.  The `ramp_dscr_ok` check is passed.

* **`ramp_dscr_ok(dscr_series)`**
    * **Logic:** This models a realistic, tiered loan covenant. It checks that the DSCR (NOI / Debt Payment) is above the minimum threshold for Year 1, Year 2, and all subsequent years, as defined in the `DSCR_RAMP` tuple.

## 4. Core Financial Utilities (`Cell 1`)

These are the low-level, self-contained math functions.

* **`pmt(rate, nper, pv)`:** Standard financial formula to calculate a fixed loan payment.
* **`npv(rate, cf)`:** Standard formula to calculate Net Present Value.
* **`irr_robust(cf, ...)`:** A critical function that ensures a valid rate of return is always found. It has a fallback hierarchy:
    1.  First, it tries the fast, standard `numpy_financial.irr`.
    2.  If that fails, it falls back to a custom, stable `irr_bisect` solver.
    3.  If that *still* fails, it falls back to `mirr` (Modified IRR).
* **`irr_bisect(cf, ...)`:** A custom binary-search solver that finds the discount rate at which `npv(rate, cf) = 0`.
* **`mirr(cf, ...)`:** Calculates the Modified Internal Rate of Return, which uses separate rates for borrowing (financing) and earning (reinvestment).
